

### 🚀 Способы загрузки медиафайлов на сервер

Вот три основных метода, которые подойдут для вашей задачи. Выбирайте тот, который кажется вам удобнее.

#### 1. Использование SCP (Secure Copy) — просто и надежно
Это классический и очень надежный способ для разовой загрузки или переноса больших папок. Он работает через тот же протокол, что и SSH.

*   **Как это работает:** Вы используете команду в терминале вашего локального компьютера, которая копирует файлы по защищенному каналу.
*   **Когда использовать:** Идеально для первоначальной заливки всех медиафайлов на сервер.
*   **Команда для копирования папки `assets` целиком:**
    ```bash
    scp -P 62965 -r C:\Users\SDL\Desktop\SDLWebSite\assets root@93.88.203.89:/var/www/html/
    ```
    Разберем ключи:
    *   `-P 62965` — указывает на нестандартный SSH-порт вашего сервера. Без него команда будет искать порт 22 по умолчанию.
    *   `-r` — рекурсивный режим, необходим для копирования папок со всем их содержимым.
    *   `C:\Users\...\assets` — путь к папке с медиафайлами на вашем локальном компьютере.
    *   `root@93.88.203.89:/var/www/html/` — адрес и папка назначения на сервере.
*   **Дополнительный совет:** Если в будущем понадобится скопировать только новые или измененные файлы, можно использовать флаг `-u` (update).

#### 2. Синхронизация с Rsync — для профи
Если вы планируете часто обновлять медиатеку на сервере, `rsync` будет гораздо удобнее. Он копирует **только измененные файлы**, что экономит время и трафик при повторных запусках.

*   **Как это работает:** Утилита сравнивает файлы в локальной и удаленной папках и переносит только различия.
*   **Когда использовать:** Для регулярного обновления больших коллекций музыки и изображений.
*   **Установка:** Rsync может быть не установлен на вашем локальном компьютере с Windows. Проще всего использовать его через **WSL (Windows Subsystem for Linux)**.
*   **Команда для синхронизации:**
    ```bash
    rsync -avz -e "ssh -p 62965" /путь/к/локальной/папке/assets/ root@93.88.203.89:/var/www/html/assets/
    ```
    *   `-a` (archive) — сохраняет права доступа, временные метки и копирует рекурсивно.
    *   `-v` (verbose) — показывает подробный отчет о процессе.
    *   `-z` — сжимает данные при передаче для ускорения.
    *   `-e "ssh -p 62965"` — указывает ssh-соединение через нестандартный порт.

#### 3. Загрузка через Git (и Git LFS) — технологично
Вы уже настроили Git-репозиторий. Медиафайлы — это бинарные данные, и Git не очень хорошо умеет с ними работать. Обычно их добавляют в `.gitignore`, но если вы хотите хранить их в репозитории, нужно использовать Git Large File Storage (LFS).

*   **Как это работает:** Git LFS заменяет большие файлы (например, MP3 и JPG) текстовыми ссылками, а сами файлы хранит отдельно.
*   **Когда использовать:** Если для вас критично хранить все файлы проекта в одном месте и управлять версиями даже изображений и музыки.
*   **Настройка:**
    1. Установите Git LFS на локальный компьютер.
    2. В репозитории выполните:
    ```bash
    git lfs track "*.mp3" "*.jpg" "*.png"
    ```
    3. Добавьте `.gitattributes` в Git и сделайте коммит.
    4. После этого вы можете добавлять и пушить медиафайлы как обычно.

### 🔧 Настройка прав доступа на сервере (важный шаг)

Какой бы способ вы ни выбрали, после загрузки файлов нужно дать веб-серверу (Nginx) и PHP разрешение на их чтение. Для этого используются команды `chown` (смена владельца) и `chmod` (смена прав).

Подключитесь к серверу по SSH и выполните:

```bash
cd /var/www/html
chown -R www-data:www-data assets
chmod -R 755 assets
```

Что здесь происходит:
*   `chown -R www-data:www-data assets` — рекурсивно (`-R`) меняет владельца папки `assets` на пользователя `www-data` и группу `www-data`. Это именно тот пользователь, от имени которого обычно работает веб-сервер.
*   `chmod -R 755 assets` — устанавливает права доступа: у владельца (`www-data`) есть все права на чтение, запись и выполнение (7), а у группы и остальных — только на чтение и выполнение (5). Это стандартные и безопасные права для файлов сайта.

### ⚙️ Настройка Nginx для отдачи статики (если потребуется)

Обычно Nginx прекрасно отдает статические файлы (изображения, музыку) без дополнительных настроек, если они лежат в папке `root`, указанной в конфигурации. В вашей конфигурации это `root /var/www/html;`, так что доступ к `/assets/` уже должен быть.

Однако для обеспечения безопасности и предотвращения просмотра содержимого папок, где нет индексного файла, добавьте в конфиг сайта (`/etc/nginx/sites-available/sdltech.tech`) следующие строки:

```nginx
location ~ ^/assets/(Music|Pictures)/ {
    autoindex off;
}
```

После внесения изменений не забудьте проверить конфигурацию и перезагрузить Nginx:

```bash
nginx -t
systemctl reload nginx
```

### 💎 Итог и рекомендации

Вот краткий план действий, который объединяет все шаги:

1.  **Первичная загрузка:** Используйте команду `scp -P 62965 -r` из первого способа, чтобы скопировать всю папку `assets` на сервер.
2.  **Настройка прав:** Подключитесь к серверу и выполните команды `chown -R www-data:www-data assets` и `chmod -R 755 assets`.
3.  **Проверка:** Откройте в браузере адрес вашего сайта и убедитесь, что изображения и музыка загружаются.
4.  **Безопасность:** Добавьте в конфиг Nginx правило `autoindex off;` и перезагрузите его.
5.  **Будущие обновления:** Если медиатека будет часто меняться, настройте `rsync` (способ 2). Это сэкономит время при каждой синхронизации.
